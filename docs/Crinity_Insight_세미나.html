01 · Why

왜 만들었나 — 파이프라인화 된 가시성

4개 솔루션이 각각 독립 동작해, 메일 한 통 처리 전방을 아무도 하나로 보지 못했다. 같은 원인이 두 방향의 비용으로 흘러들어온다.

고객사 — CS 티켓 폭증

  • 메일 지연·미도달 시 어느 장비에서 막혔는지 알 수 없음
  • 차단 사유(스팸? DLP? 정책?)를 확인할 방법이 없어 즉시 고객콜센터 문의
  • "내 메일 어디 갔나요" 질의가 CS의 절반 이상

엔지니어 — MTTR 지연

  • 장애 원인 파악 시 4개 장비에 각각 SSH 접속해 grep 교차 검증
  • 로그 수집만도 장비당 10분, 총 40분 이상 소요
  • 수기의 수집 과정은 누락·오탈 등 휴먼 에러 의존
→ 단순 로그 뷰어가 아니라, 가시성 파이프라인을 해결하는 "메타 관제 + 지속보안 자동화 엔진"이 필요했다.
02 · Constraints

모든 기술 선택을 결정한 3가지 제약

기술 스택은 여기서 드러난다 — 아래 3개 제약을 통과하지 못하는 선택지는 처음부터 후보가 아니었다.

폐쇄망 고객사

공공·금융 고객 다수. 외부 SaaS·인터넷 저장소 접근 불가. 고객사 DB가 이미 고정돼 있어 별도 DB 엔진을 들이거나 새 포트(예: PostgreSQL)를 열기도 어렵다.

  • Datadog류 SaaS APM 배제
  • OS 내장 도구(Python 3 · Bash · tar · top · df)만 조합
  • DB는 SQLite 채택 — 별도 서버·포트 없이 파일 1개로 영속화(리스크 체크리스트 "DB 선정" 결론)
  • 다중 고객사·서버 구축 시 사이트 식별(INSIGHT_SITE_NAME/ID) 배지로 "지금 어느 사이트인지" 혼동 방지

상용 장비 Zero-Impact

4개 솔루션은 이미 운영 중인 상용 서비스. 사이드카·패키지 설치 최소화.

  • 무거운 수집기(Filebeat + ES) 배제
  • 초경량 자체 Agent — CPU < 5%, MEM < 100MB

도메인 특화 데이터

Message-ID로 이어지는 "메일 여정"이라는 도메인 개념이 핵심.

  • 범용 로그 검색이 아닌 여정 Join 엔진 직접 구현
  • 격리 해제 등 조치까지 솔루션 API 연동
발화점 재발명 없이 — 도메인 로직(Join · Timeout · 룰)과 조치 연동 · 신속 UX만 직접 만든다.

04 · Core Modules

핵심 기능 3대 모듈

항상 똑같은 모듈별 What / How / Result의 로직 분해를 확인하세요.

WHAT

독립 서버를 통과하는 개별 메일의 상태(수신·DLP 검사·스팸 필터·보관)를 하나의 UI에서 진행철 노선도 타임라인으로 시각화.

HOW

Edge Agent가 정규식으로 Message-ID·상태만 추출 → 중앙서버가 Message-ID를 PK로 4개 서버 데이터를 Join·캐싱 → React가 경로·지연시간 렌더링.

RESULT

고객사가 스스로 차단 사유 확인 → CS 티켓 50%↓. 구축한 로그 추적 20분 → 10초.

WHAT

장애 시 즉시 4개 서버의 로그·리소스 상태를 원클릭으로 동시 수집·압축하는 자동화 도구.

HOW

시간대 지정 → 중앙서버 비동기 fan-out → Agent×4 병렬 수집(로그·top·df) → 노드별 압축 업로드 → 중앙 병합 후 단일 파일 다운로드(manifest 동봉).

RESULT

장비당 10분 × 4대 = 40분+ 소요였던 것을 1분 이내로. 휴먼 에러 원천 차단.

RUNNING4개 노드 병렬 수집 진행
DONE전 노드 성공 · 병합 완료
PARTIAL_DONE일부 실패 — 성공분만 병합, 실패 노드 명시
FAILED전 노드 실패 · 재시도 안내

M2-F9 · 저장 경로 설정 — 입력 검증 6케이스

경로 값은 그대로 mkdir·tar 인자로 들어간다 — 명령 주입·Path Traversal의 최우선 표적. 화이트리스트 위주로 방어한다.

케이스입력 예판정근거
정상 절대 경로/data/insight/sos통과절대 경로 + 허용 문자만 사용
상대 경로data/sos거부절대 경로 강제 — 실행 위치 의존 차단
시스템 디렉토리/etc/sos거부금지 디렉토리 목록(/etc, /boot 등)
Path Traversal/data/../etc거부'..' 포함 시 즉시 거부
명령 주입 문자/data;rm -rf거부화이트리스트 [A-Za-z0-9/_.-] 외 전부 거부
비허용 문자(한글 등)/데이터/sos거부화이트리스트 위치 — 예외를 두지 않는다
클라이언트 검증은 UX용일 뿐 — 서버측 동일 규칙 + 실제 쓰기·여유공간 검사(statvfs) + systemd ReadWritePaths 등록이 실질 방어선. 변경은 ADMIN 권한 + 감사 기록, 진행 중 job에는 미적용.

WHAT

장비 다운 전 메일 큐 적체·디스크 한계를 먼저 감지해 알림을 발송하는 사전 경보 체계.

HOW

Agent 10초 주기 지표 수집 → 룰 평가 → 쿨다운·중복 제거 → Slack/Teams Webhook 발송 → 미조치 시 에스컬레이션 → 정상 복귀 시 RESOLVED 자동 통보.

RESULT

고객이 장애를 인지하기 전에 크리니티가 먼저 움직이는 능동적(Proactive) 지속보안 체계로 격상.

룰 예시 — 문턱값을 잡는다

  • MailBreaker 큐 1,000건 초과 대기 → CRITICAL
  • 디스크 사용률 90% 초과 → WARNING
  • CPU 지속 임계 초과 → WARNING (지속시간 조건)
  • Agent 하트비트 유실 → Watchdog가 노드 다운 여부 경보
  • 장비 IP·Port 연결 끊김/복구(신규) → CRITICAL/INFO — 관리자가 등록한 주소로 실TCP 연결 재확인 실패 시

알림 폭주 막는 장치

  • 쿨다운 — 동일 경보 반복 발송 억제
  • 중복 제거 — 같은 원인의 경보 병합
  • 에스컬레이션 — 미조치 시 단계 상향
  • RESOLVED — 정상 복귀를 자동으로 알려 잊음 정리

06 · Code Review

로직 검증 로드맵 — 어디부터 읽을 것인가

프로토타입(crinity_insight_admin.html)은 §1~§9 섹션 주석 체계로 구성. 읽는 순서는 → UI보다 데이터 계약과 상태 전이를 먼저 본다.

§1 DATA데이터 계약

목업 상수의 필드명·구조가 API 스키마(설계서 §4.3)와 1:1인지 확인. 이후 모든 로직의 전제.

§4 SCR-02Join · Timeout 엔진

여정 병합의 멱등성, 60초 Timeout 판정, 상태 전이(OK→QUARANTINED→…)가 도메인 규칙과 일치하는지. 플랫폼의 심장.

§5 SCR-03SOS 상태 머신

RUNNING→DONE / PARTIAL_DONE / FAILED 전이와 부분 실패 처리, M2-F9 경로 검증 6케이스.

§6 SCR-04경보 룰

임계치 평가·쿨다운·에스컬레이션·RESOLVED 조건. 오탐/미탐 경계가 핵심.

§3 · §9관제 갱신 주기

3초 폴링 갱신 흐름과 라이브 시뮬레이션 타이머. 실구현 시 fetch로 치환되는 지점.

§8 SCR-06전환 체크리스트

MOCK→REAL 전환 포인트(폴링·SSE·RBAC·감사 로그)가 빠짐없이 목록화됐는지 최종 대조.
